# Internationalization (i18n)

## Overview

Yusr Academy uses **next-intl 4.12.0** for server-side internationalization with support for **Arabic** (default) and **English**.

---

## Configuration

### Routing Configuration

**File**: `i18n/routing.ts`

```typescript
import { defineRouting } from "next-intl/routing";

export const routing = defineRouting({
  locales: ["ar", "en"],
  defaultLocale: "ar",
});
```

### Request Configuration

**File**: `i18n/request.ts`

```typescript
import { getRequestConfig } from "next-intl/server";
import { routing } from "./routing";

export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as "ar" | "en")) {
    locale = routing.defaultLocale;
  }

  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

### Middleware

**File**: `middleware.ts`

```typescript
import createMiddleware from "next-intl/middleware";
import { routing } from "./i18n/routing";

export default createMiddleware(routing);

export const config = {
  matcher: ["/((?!api|_next|_vercel|.*\\..*).*)"],
};
```

**What it does**:
- Detects locale from URL path (e.g., `/ar/...`, `/en/...`)
- Redirects to default locale if no locale in path
- Sets Accept-Language header
- Does NOT handle authentication (that's in layouts)

## Usage Patterns

### Server Components

```typescript
import { useTranslations } from "next-intl";

export default function Page() {
  const t = useTranslations("student.dashboard");
  
  return (
    <div>
      <h1>{t("title")}</h1>
      <p>{t("upcomingSessions")}</p>
    </div>
  );
}
```

### Client Components

```typescript
"use client";

import { useTranslations } from "next-intl";

export function Component() {
  const t = useTranslations("common");
  
  return <button>{t("submit")}</button>;
}
```

### With Interpolation

```typescript
const t = useTranslations("student.progress");

// Message: "لقد أتممت {count} جزء"
t("juzCompleted", { count: 5 }); // "لقد أتممت 5 جزء"
```

### With Pluralization

```typescript
const t = useTranslations("common");

// Messages:
// "badge": "وسام",
// "badges": "أوسمة"

t("badge", { count: 1 }); // "وسام"
t("badge", { count: 5 }); // "أوسمة"
```

### Server Actions

Server Actions typically return i18n keys, not translated strings:

```typescript
"use server";

export async function action(formData: FormData) {
  // Validation failed
  return { error: "validationError" };
  
  // Email already exists
  return { error: "emailExists" };
  
  // Success
  return { success: true };
}
```

Client component translates the error:

```typescript
"use client";

export function Form() {
  const [state, formAction] = useActionState(action, null);
  const t = useTranslations("validation");
  
  return (
    <form action={formAction}>
      {state?.error && <p>{t(state.error)}</p>}
    </form>
  );
}
```

## Locale Switching

### User Preference Storage

User's preferred locale is stored in `User.locale` field (default: "ar").

### Switching Locale

Locale switching is done via URL path change:

```typescript
import { useRouter, usePathname } from "next/navigation";
import { useLocale } from "next-intl";

export function LocaleSwitcher() {
  const router = useRouter();
  const pathname = usePathname();
  const currentLocale = useLocale();
  
  const switchLocale = (newLocale: string) => {
    // Replace current locale in path
    const newPathname = pathname.replace(`/${currentLocale}`, `/${newLocale}`);
    router.push(newPathname);
  };
  
  return (
    <button onClick={() => switchLocale(currentLocale === "ar" ? "en" : "ar")}>
      {currentLocale === "ar" ? "English" : "العربية"}
    </button>
  );
}
```

## RTL/LTR Support

### Automatic Direction

The locale's text direction is automatically applied via the `dir` attribute:

```typescript
// app/[locale]/layout.tsx
import { routing } from "@/i18n/routing";

export default async function Layout({ children, params }) {
  const { locale } = await params;
  const direction = locale === "ar" ? "rtl" : "ltr";
  
  return (
    <html lang={locale} dir={direction}>
      <body>{children}</body>
    </html>
  );
}
```

### Tailwind RTL Utilities

Tailwind CSS automatically flips certain utilities in RTL mode:

- `ml-4` becomes margin-right in RTL
- `pl-4` becomes padding-right in RTL
- `text-left` becomes text-right in RTL

For explicit directional control:

```tsx
<div className="ltr:ml-4 rtl:mr-4">Content</div>
```

## Date & Number Formatting

### Using Intl APIs

```typescript
const locale = useLocale();

// Date formatting
const date = new Date();
const formatted = new Intl.DateTimeFormat(locale, {
  dateStyle: "long",
}).format(date);

// Number formatting
const number = 12345.67;
const formattedNumber = new Intl.NumberFormat(locale).format(number);

// Currency formatting
const price = 99.99;
const formattedPrice = new Intl.NumberFormat(locale, {
  style: "currency",
  currency: "SAR",
}).format(price);
```

### Arabic Numerals

Arabic locale uses Eastern Arabic numerals (٠١٢٣٤٥٦٧٨٩) by default when formatted via Intl APIs.

For Western Arabic numerals (0123456789), use:

```typescript
const formatted = new Intl.NumberFormat("ar-u-nu-latn").format(number);
```

## Message Organization Best Practices

### Namespace by Feature

Group related messages under a feature namespace:

```json
{
  "student": {
    "dashboard": { ... },
    "profile": { ... },
    "assignments": { ... }
  }
}
```

### Reuse Common Messages

Extract commonly used messages to `common` namespace:

```json
{
  "common": {
    "save": "حفظ",
    "cancel": "إلغاء",
    "delete": "حذف",
    "edit": "تعديل"
  }
}
```

### Use Descriptive Keys

Use descriptive keys that indicate purpose:

```json
{
  "student": {
    "dashboard": {
      "welcomeMessage": "مرحباً",
      "noUpcomingSessions": "لا توجد جلسات قادمة"
    }
  }
}
```

### Avoid Nesting Too Deep

Limit nesting to 2-3 levels for maintainability:

```json
// Good
{
  "student": {
    "dashboard": {
      "title": "لوحة الطالب"
    }
  }
}

// Avoid
{
  "student": {
    "views": {
      "dashboard": {
        "header": {
          "title": "لوحة الطالب"
        }
      }
    }
  }
}
```

## Server vs Client Translation

### Server Components (Recommended)

```typescript
import { useTranslations } from "next-intl";

export default function Page() {
  const t = useTranslations("namespace");
  return <div>{t("key")}</div>;
}
```

**Pros**:
- No JavaScript sent to client for translations
- Better performance
- SEO-friendly

### Client Components

```typescript
"use client";

import { useTranslations } from "next-intl";

export function Component() {
  const t = useTranslations("namespace");
  return <div>{t("key")}</div>;
}
```

**Cons**:
- Entire message JSON sent to client
- Larger bundle size

## Missing Translation Handling

If a translation key is missing, next-intl will:

1. Return the key itself in development (e.g., "student.dashboard.missingKey")
2. Log a warning in the console
3. In production, return the key or a fallback message

To provide a fallback:

```typescript
t("missingKey", {}, { default: "Fallback text" })
```

## Performance Considerations

### Message Loading

- Messages are loaded per-request (not bundled)
- Only the current locale's messages are loaded
- Messages are cached in memory

### Bundle Size

- Server Components: 0 KB to client (translations stay on server)
- Client Components: Full message JSON sent to client (~50-100 KB)

### Optimization Tips

1. Prefer Server Components for static content
2. Use Client Components only when interactivity requires it
3. Split large namespaces into smaller ones
4. Remove unused translation keys

## Common Patterns

### Form Validation Errors

```typescript
// Server Action
return { error: "emailExists" };

// Client Component
const t = useTranslations("validation");
{state?.error && <p>{t(state.error)}</p>}

// messages/ar.json
{
  "validation": {
    "emailExists": "البريد الإلكتروني مسجل بالفعل",
    "passwordTooShort": "كلمة المرور قصيرة جداً"
  }
}
```

### Dynamic Content

```typescript
const t = useTranslations("student.progress");

// Message: "لقد أتممت {juz} جزء من أصل {total}"
t("progress", { juz: 5, total: 30 });
```

### Pluralization

```typescript
const t = useTranslations("common");

// messages/ar.json
{
  "common": {
    "students": {
      "zero": "لا طلاب",
      "one": "طالب واحد",
      "two": "طالبان",
      "few": "{count} طلاب",
      "many": "{count} طالباً",
      "other": "{count} طالب"
    }
  }
}

t("students", { count: 0 }); // "لا طلاب"
t("students", { count: 1 }); // "طالب واحد"
t("students", { count: 2 }); // "طالبان"
t("students", { count: 5 }); // "5 طلاب"
```

## Hardcoded Locale Issues

**Known Issue**: Some redirects in Server Actions are hardcoded to `/ar/`:

```typescript
// server/actions/auth.ts
redirect("/ar/login"); // Should use current locale
```

**Fix** (pending implementation):

```typescript
import { getLocale } from "next-intl/server";

const locale = await getLocale();
redirect(`/${locale}/login`);
```

This affects:
- `loginAction` redirects
- `logoutAction` redirect
- `revalidatePath` calls in mutations
