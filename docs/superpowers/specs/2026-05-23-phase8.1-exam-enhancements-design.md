# Phase 8.1: Exam Enhancements вЂ” Design Spec

## Goal

Enhance the existing exam system with five features: per-exam time limits, question randomization, retakes with best-score logic, tag-based question pools, and template duplication. All changes extend the current schema and UI incrementally вЂ” no new models or feature flags required.

## Scope

**In scope:**
- Time limits (per-exam countdown, server-enforced)
- Question randomization (shuffle order per student per attempt)
- Retakes (multiple attempts, best score wins, configurable max)
- Question pools (tag-based random selection from template questions)
- Exam duplication (deep-clone template + questions)

**Out of scope (deferred):**
- Audio recording for recitation questions
- Bulk CSV import of questions
- Per-question time limits
- Exam analytics (covered by Phase 9)

## Architecture

All five features extend the existing exam models (`ExamInstance`, `ExamSubmission`, `ExamQuestion`) with new optional fields. No new database models. The service layer (`server/services/exam.ts`) gains new logic for pool selection, shuffle, retake management, and duplication. The student exam-taking page gets a `CountdownTimer` client component.

### Key principle

Every new field is optional and defaults to current behavior. Existing exams continue to work without modification: no time limit, no shuffle, single attempt, all questions, no tags.

## 1. Schema Changes

### ExamInstance вЂ” 4 new fields

```prisma
model ExamInstance {
  // ... existing fields ...
  timeLimitMinutes  Int?      // null = no limit
  shuffleQuestions  Boolean   @default(false)
  maxAttempts       Int?      // null = single attempt (backward compat)
  poolConfig        Json?     // e.g. {"pick": 10, "tags": ["tajweed"]}
}
```

- `timeLimitMinutes`: Whole minutes, minimum 1. Null means untimed.
- `shuffleQuestions`: When true, each submission gets a randomized question order.
- `maxAttempts`: Null or 1 = single attempt. 2+ = multiple allowed. Max value: 10.
- `poolConfig`: JSON object with `pick: number` (required) and optional `tags: string[]`. Null means use all template questions in default order.

### ExamSubmission вЂ” 2 changes

```prisma
model ExamSubmission {
  // ... existing fields ...
  attemptNumber  Int    @default(1)
  questionOrder  Json?  // array of question IDs in display order

  // CHANGE: unique constraint updated
  @@unique([instanceId, studentId, attemptNumber])  // was @@unique([instanceId, studentId])
}
```

- `attemptNumber`: Starts at 1, increments per retake. Default 1 preserves existing rows.
- `questionOrder`: Array of question ID strings. Generated at submission creation time when shuffle or pool is active. Null means use template's default `order` field.

### ExamQuestion вЂ” 1 new field

```prisma
model ExamQuestion {
  // ... existing fields ...
  tags  String[]  // e.g. ["tajweed", "juz-1"]
}
```

- `tags`: Empty array by default. Used for pool-based selection filtering.

## 2. Time Limits

### Flow

1. **Assignment:** Moderator sets `timeLimitMinutes` when assigning exam to groups (optional number input in assign form).
2. **Pre-start:** Student sees the time limit on their exam list and on the exam page before starting.
3. **Start:** When student begins (clicks "Start Exam"), `startedAt` is set on `ExamSubmission` (already exists). Countdown begins.
4. **During exam:** `CountdownTimer` client component shows remaining time in a sticky bar. Calculates `startedAt + timeLimitMinutes - now()`.
5. **Expiry:** When timer reaches zero, component auto-submits current answers and shows "Time expired" message.
6. **Return:** If student navigates away and returns, timer resumes from `startedAt` (server-authoritative).

### Server enforcement

`saveAnswers()` rejects submissions when `startedAt + timeLimitMinutes + 30 seconds` has passed. The 30-second grace period accounts for network latency during auto-submit. Already-submitted exams are unaffected.

### CountdownTimer component

- Path: `components/exam/countdown-timer.tsx`
- `"use client"` component
- Props: `endTime: Date`, `onExpire: () => void`
- Displays `MM:SS` format
- Visual: normal text when > 5 min, amber when в‰¤ 5 min, red when в‰¤ 1 min
- Calls `onExpire` callback when timer hits zero

## 3. Question Randomization

### Flow

1. Moderator enables `shuffleQuestions` toggle when assigning exam.
2. When `getOrCreateSubmission()` creates a new submission:
   - Fetches all question IDs for the template (or the pool-selected subset)
   - Generates a Fisher-Yates shuffle of the IDs
   - Stores the permutation in `questionOrder: Json`
3. Student exam page renders questions in `questionOrder` sequence.
4. Each student gets a different order. Each retake attempt gets a fresh shuffle.
5. If `shuffleQuestions` is false and no pool is active, `questionOrder` is null and questions render by the default `order` field.

### Grading

Moderator grading page always shows questions in the student's `questionOrder` вЂ” they see what the student saw.

## 4. Retakes

### Configuration

- `maxAttempts` on `ExamInstance`: null = single attempt (backward compatible), 2вЂ“10 = that many attempts allowed.
- Set by moderator in the assign form (optional number input).

### Submission lifecycle

1. Student completes attempt 1 в†’ submission status becomes SUBMITTED в†’ moderator grades в†’ GRADED.
2. If `attemptNumber < maxAttempts` and current attempt is GRADED, student sees "Retake" button.
3. Clicking "Retake" calls a new `createRetakeSubmission()` function:
   - Creates new `ExamSubmission` with `attemptNumber = previous + 1`
   - Generates fresh `questionOrder` if shuffle/pool is active
   - Status starts at NOT_STARTED
4. Process repeats until `attemptNumber >= maxAttempts`.

### Best score logic

- New service function: `getBestSubmission(instanceId, studentId)` вЂ” returns the graded attempt with the highest `totalScore`.
- `getStudentInstances()` returns best score per instance for the student exam list.
- Admin results page and analytics use best score.
- All attempts remain visible in the moderator grading view.

### UI changes

- **Student exam list:** Shows best score and attempt count (e.g. "82% (2/3 attempts)").
- **Student exam page:** After grading, shows "Retake Exam" button if attempts remain. Shows attempt history below.
- **Moderator instance detail:** Submissions table shows all attempts. Best score highlighted. Column for attempt number.
- **Admin results:** Shows best score with attempt indicator.

## 5. Question Pools (Tag-Based)

### Tag management

- When adding/editing a question, moderator can add tags via a text input with chip/pill UI.
- Tags are free-text, lowercased, trimmed. Stored as `String[]` on `ExamQuestion`.
- Tags display as badges in the question list on the admin exam detail page.

### Pool configuration

- When assigning exam to groups, moderator can optionally configure a pool:
  - "Pick N questions" вЂ” number input
  - "Filter by tags" вЂ” optional, multi-select from tags that exist on the template's questions
- Stored as `poolConfig: Json` on `ExamInstance`: `{"pick": 10, "tags": ["tajweed"]}` or `{"pick": 15}` (no tag filter).

### Selection logic

In `getOrCreateSubmission()`, when `poolConfig` is present:

1. Fetch all questions for the template.
2. If `poolConfig.tags` is set, filter to questions that have at least one matching tag.
3. Randomly select `poolConfig.pick` questions from the filtered set.
4. Store selected IDs in `questionOrder` (shuffled if `shuffleQuestions` is also on, otherwise in original `order`).
5. Calculate effective `totalPoints` from selected questions only.

### Validation

At assignment time (`assignToGroups()`):
- Count questions matching the tag filter.
- If `pick > matchingCount`, reject with error: "Not enough questions matching the selected tags. Found {matchingCount}, need {pick}."
- If no `poolConfig`, all questions are used as before.

### Per-attempt variation

Each retake attempt gets a fresh random selection from the pool. Students may see different questions across attempts.

## 6. Exam Duplication

### Service function

`duplicateTemplate(templateId: string, actorId: string)`:

1. Fetch template with all questions.
2. Create new `ExamTemplate` with same fields, title suffixed with " (copy)".
3. Deep-clone all `ExamQuestion` rows with new IDs, preserving order, tags, and all fields.
4. Return the new template.
5. Audit log: "Duplicated exam template {originalTitle}".

### UI

- Admin exam detail page: "Duplicate" button next to the existing edit actions.
- After duplication, redirect to the new template's detail page.
- Server action: `duplicateTemplateAction(formData: FormData)` вЂ” extracts `templateId`, calls service, revalidates paths.

## 9. Service Layer Changes

### `server/services/exam.ts` modifications

**Modified functions:**
- `assignToGroups()` вЂ” accept and validate new fields (timeLimitMinutes, shuffleQuestions, maxAttempts, poolConfig)
- `getOrCreateSubmission()` вЂ” handle pool selection, shuffle, and store questionOrder
- `saveAnswers()` вЂ” enforce time limit server-side
- `getStudentInstances()` вЂ” return best score per instance
- `getInstanceDetail()` вЂ” include new fields and all attempts per student
- `addQuestion()` вЂ” accept tags field
- `getAllInstances()` вЂ” include attempt counts in results

**New functions:**
- `createRetakeSubmission(instanceId, studentProfileId)` вЂ” create new attempt
- `getBestSubmission(instanceId, studentId)` вЂ” return highest-scoring graded attempt
- `duplicateTemplate(templateId, actorId)` вЂ” deep clone template + questions

### `server/actions/exam.ts` additions

- `duplicateTemplateAction(formData)` вЂ” calls `duplicateTemplate()`
- `createRetakeAction(formData)` вЂ” calls `createRetakeSubmission()`

### `lib/validations/exam.ts` updates

- `assignToGroupsSchema` вЂ” add timeLimitMinutes, shuffleQuestions, maxAttempts, poolConfig fields
- `addQuestionSchema` вЂ” add tags field
- New `duplicateTemplateSchema` вЂ” templateId only
- New `createRetakeSchema` вЂ” instanceId only
